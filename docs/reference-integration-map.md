# 参考项目整合地图

状态：活文档
最近更新：2026-07-02

本文档把三个外部开源项目作为长期架构资产管理。Money_Never_sleep 不直接整体 fork，但允许按模块借鉴、复制、适配和重设计；每次相关计划都要写清楚来源、取舍和验证口径。

## 使用规则

- 做计划时必须检查本文件：如果新阶段涉及 AI 投研、数据源、报告、桌面体验或图表，应明确参考哪个项目。
- 可以复制成熟代码片段，但必须先确认许可证、依赖边界、输入输出契约和测试覆盖；复制后要包在 Money_Never_sleep 自己的领域模型后面。
- 不做“混合三套架构”的硬拼接。外部项目能力进入本仓库时，优先通过 adapter、provider、runner 或 UI 组件边界接入。
- 每次吸收参考项目能力后，更新“当前接入状态”和“下一步”。

## 当前优先顺序

1. 先吸收 `TradingAgents-astock` 的深度投研能力：事件流、证据、风险约束必须真正进入真实引擎。
2. 再吸收 `daily_stock_analysis` 的持续运行能力：调度、报告管理、通知和数据 fallback。
3. 最后吸收 `go-stock` 的桌面和图表体验：在研究结果可信后改善阅读和操作效率。

## 已采纳的长期补充

- 投资计划输出契约需要单独设计：每份报告最终应包含方向、仓位、买入条件、卖出条件、止损、止盈、观察窗口和复盘条件。
- 数据可信度需要量化：来源、时间、完整性、是否 fallback、是否真实引擎消费过，都应进入报告或诊断。
- 模型/引擎成本需要治理：`auto` 不只要显示命中情况，还应逐步记录耗时、失败原因、token/调用成本和回退路径。
- 合规边界必须持续可见：第一阶段只做投研建议和风险计划，不做真实下单，不伪装成确定性收益承诺。
